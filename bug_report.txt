Detailed Bug ReportsBUG-001: Real Student PII Exposed in APK AssetsSeverityCriticalCategorySecurityFile / Locationfrontend/assets/data/json/DescriptionStudent names and PIN numbers from multiple batches (2023–2028) are bundled directly into the APK as JSON asset files (alwar_students.json, 2023-2026_students.json, etc.). Anyone who extracts the APK can read all student records without authentication.Recommended FixRemove all PII from bundled assets. Fetch student lists from the backend API with proper authentication. If offline access is needed, encrypt the data before bundling.BUG-002: Permissive CORS Enabled in Production BackendSeverityCriticalCategorySecurityFile / Locationbackend/src/main.rsDescriptionThe server uses CorsLayer::permissive() which allows requests from any origin with any headers and methods. This exposes all API endpoints to cross-origin attacks from any website.Recommended FixReplace CorsLayer::permissive() with a strict CORS policy that whitelists only the app&apos;s known origins. Example: CorsLayer::new().allow_origin([...])BUG-003: Production Server Panics on Bind/Serve FailureSeverityCriticalCategoryCrash / Error HandlingFile / Locationbackend/src/main.rsDescriptionThe server uses .unwrap() on both TcpListener::bind() and axum::serve(). If the port is already in use or there is any socket error, the server will immediately panic and crash with no recovery or meaningful error reporting.Recommended FixReplace .unwrap() with proper error handling using .expect() or match. Log the error clearly and exit with a non-zero code so the process manager can detect the failure.BUG-004: Debug println! Statements Left in Production CodeSeverityHighCategoryPerformance / SecurityFile / Locationbackend/src/routes/student.rs, main.rs, othersDescriptionDozens of println!(&quot;DEBUG: ...&quot;) statements are present throughout the Rust backend, including ones that print SQL queries, branch names, and user IDs. These leak internal logic and data to server logs in production.Recommended FixReplace all println!(&quot;DEBUG:...&quot;) with the tracing crate (tracing::debug!) and gate them behind the DEBUG log level. Ensure production deployments use the INFO or WARN level.BUG-005: Duplicate Debug Log StatementSeverityHighCategoryCode QualityFile / Locationbackend/src/routes/student.rs (get_student_courses_handler)DescriptionThe same debug log line is printed twice consecutively: println!(&quot;DEBUG: Fetching courses for Branch=&apos;{}&apos;, Semester=&apos;{}&apos;, Section=&apos;{}&apos;&quot;). This indicates a copy-paste error and doubles log noise.Recommended FixRemove the duplicate println! statement. Only one log line is needed per request.BUG-006: Course Credits Calculated from Subject Name LengthSeverityHighCategoryLogic BugFile / Locationbackend/src/routes/student.rs (get_student_courses_handler)DescriptionCredits are computed as: let credits = if s.name.len() % 2 == 0 { 4 } else { 3 }. This is based on the character count of the subject name, not actual curriculum data. A subject named &quot;Math&quot; (4 chars, even) gets 4 credits; &quot;Physics&quot; (7 chars, odd) gets 3. This is completely incorrect.Recommended FixAdd a credits column to the subjects table in the database. Use the actual stored credit value when returning course information.BUG-007: Lesson Plan Progress Expectation is HardcodedSeverityHighCategoryLogic BugFile / Locationbackend/src/routes/student.rs (get_student_courses_handler)DescriptionThe expected progress threshold is hardcoded as let expected_progress = 30 with a comment &apos;Mock expectation for now&apos;. This means every course is judged against a fixed 30% baseline regardless of the actual academic calendar or syllabus plan.Recommended FixCalculate expected_progress dynamically based on how far through the semester the current date is, or store per-subject expected milestones in the database.BUG-008: Branch Normalization Mismatch Between main.rs and models.rsSeverityHighCategoryLogic Bug / Data IntegrityFile / Locationbackend/src/main.rs, backend/src/models.rsDescriptionfix_branch_names() in main.rs maps &apos;CSE&apos; -&gt; &apos;Computer Engineering&apos;, but the normalize_branch() function in models.rs does NOT handle &apos;CSE&apos;. If a user registers with &apos;CSE&apos;, fix_branch_names will update the DB, but any new data with &apos;CSE&apos; will pass through normalize_branch unchanged, causing inconsistent branch values.Recommended FixAdd &apos;CSE&apos; to the normalize_branch() match block in models.rs to ensure consistency across all code paths.BUG-009: UUID Parse Failure Falls Back to Zero UUIDSeverityHighCategoryCrash / Error HandlingFile / Locationbackend/src/routes/student.rs (get_student_lesson_plan_handler)DescriptionWhen parsing user_id as UUID, the code uses Uuid::parse_str(uid_str).unwrap_or_default(). The default UUID is all-zeros (00000000-0000-0000-0000-000000000000). If an invalid user_id is passed, the query silently runs against this zero UUID instead of returning a 400 Bad Request error.Recommended FixReplace unwrap_or_default() with proper error handling: Uuid::parse_str(uid_str).map_err(|_| (StatusCode::BAD_REQUEST, ...))?BUG-010: gRPC SignupRequest Proto Missing Critical FieldsSeverityHighCategoryAPI / Data FlowFile / Locationprotos/auth.proto, frontend/lib/core/services/grpc/DescriptionThe gRPC SignupRequest proto only defines: fullName, loginId, password, branch, year. However the REST API SignupRequest model has 12+ fields including phone_number, dob, experience, email, semester, batch_no, title, section. Any signup going through the gRPC path will silently drop all these fields.Recommended FixUpdate the auth.proto SignupRequest to include all the same fields as the REST model. Regenerate the Dart gRPC client code after updating the proto.BUG-011: fix_branch_names() Runs on Every Server StartupSeverityMediumCategoryPerformanceFile / Locationbackend/src/main.rsDescriptionThe fix_branch_names() function executes UPDATE queries against 7 database tables for 20+ branch name variations on every single server restart. On a production system with many rows, this causes unnecessary load every time the server restarts.Recommended FixRun fix_branch_names() as a one-time SQL migration instead. Once all data is normalized, remove the function call from startup.BUG-012: Section Always Defaults to &apos;Section A&apos; on Missing DataSeverityMediumCategoryLogic BugFile / Locationbackend/src/routes/student.rsDescriptionMultiple places in the backend fall back to &apos;Section A&apos; when a user&apos;s section is not found: unwrap_or_else(|| &quot;Section A&quot;.to_string()). This means students without a section assignment are silently placed in Section A for queries, potentially showing them wrong data.Recommended FixReturn an appropriate error or empty result when a student&apos;s section is not set, rather than defaulting to Section A.BUG-013: Notification Service Creates New Instance for CancelSeverityMediumCategoryLogic BugFile / Locationfrontend/lib/core/services/notification_service.dartDescriptionThe cancelNotification method creates a fresh NotificationService() instance: final service = NotificationService(); await service._notificationsPlugin.cancel(id). Log files may contain sensitive data like error messages with SQL details or user data.Recommended FixRemove all diagnostic scripts from the repository. Add *.txt, *.log, and debug_*.js to .gitignore. Move utility scripts to a separate /tools or /scripts folder if needed.BUG-015: No JWT Token Expiry or Refresh HandlingSeverityMediumCategorySecurity / AuthFile / Locationfrontend/lib/core/services/auth_service.dartDescriptionJWT tokens are stored and used for authentication, but there is no visible token refresh mechanism or expiry check. If a token expires while the user is active, API calls will fail with 401 errors without a clear user-facing message or automatic re-authentication.Recommended FixImplement a token refresh flow. Check token expiry before API calls and silently refresh using a refresh token, or redirect to the login screen with an appropriate message.BUG-016: Rapid Schema Migrations Suggest InstabilitySeverityMediumCategoryData IntegrityFile / Locationbackend/migrations/DescriptionThere are 35+ migrations with multiple same-day patches (e.g., 20260123140000 and 20260123140600 on the same minute, 20260113160000 through 20260113174500 all in the same hour).
